# CLUBHUB STATE AUTHORITY

**Version:** 1.0.0
**Status:** ENFORCED
**Authority:** Defines the canonical data authority model for ClubHub TV.
Every component must read state from the authoritative tier and write only within
its ownership boundary. Violations produce Class D or Class F failures as defined
in CLUBHUB_SYSTEM_CONTRACTS.md §2.

---

## 0. PURPOSE

ClubHub TV is a distributed system with multiple state stores: a PostgreSQL database,
a manifest cache table, screen-local disk caches, and in-memory fleet state in the
simulator and management API. Without a defined authority hierarchy, components can
serve stale, conflicting, or fabricated state. This document defines who owns what,
who can write what, and how conflicts are resolved.

---

## 1. STATE TIERS

### Tier 1 — Authoritative (PostgreSQL)

**Owner:** Backend only. No other component writes to this tier.

| Table | Contains | Write Owner | TTL |
|-------|----------|-------------|-----|
| `content` | Content items (template_type, data JSON) | Backend `/content` route | Permanent until deleted |
| `schedules` | Scheduling rules (screen/venue/time constraints) | Backend `/schedules` route | Permanent until deleted |
| `screens` | Screen registration and metadata | Backend `/screens` route | Permanent until deleted |
| `venues` | Venue metadata | Backend `/venues` route | Permanent until deleted |
| `playlists` | Pre-computed playlist by screen_id, version, checksum | Backend manifest computation | Recomputed on content/schedule change |

**Properties:**
- This is the ONLY source a backend instance may read when computing a manifest.
- Loss of Tier 1 access = backend enters DEGRADED state (`/health/ready` → 503).
- Tier 1 is the ground truth for Class F failure detection (desync, version regression).

### Tier 2 — Derived Cache (manifest_cache table)

**Owner:** Backend only. No Pi or external process writes to this tier.

| Entry | Contains | Write Owner | TTL |
|-------|----------|-------------|-----|
| `manifest_cache` row | Computed manifest JSON, version, checksum, computed_at, valid_until | Backend manifest route | 5s TTL (in-memory); invalidated on content/schedule change |

**Properties:**
- Derived entirely from Tier 1. Can always be regenerated by recomputing from Tier 1.
- A cache miss is not an error. It triggers a Tier 1 read and cache population.
- A stale cache entry (beyond TTL) MUST be recomputed from Tier 1, not served.
- Cache corruption or invalidation does NOT constitute a data loss event.
- `make clear-caches` (simulator) or a DB query clearing `manifest_cache` is safe
  at any time. The next manifest poll will cold-compute from Tier 1.

### Tier 3 — Screen-Local Disk Cache

**Owner:** Pi appliance. Backend has no visibility into this tier.

| Entry | Contains | Write Owner | TTL |
|-------|----------|-------------|-----|
| Local manifest file | Last successfully fetched manifest JSON | Pi appliance | Until replaced by a valid poll |

**Properties:**
- Read-only from the system's perspective. Written only by the Pi during a successful poll.
- Authoritative only when the Pi is offline (all Tier 1/2 unreachable).
- NEVER used to answer a manifest query if Tier 2 is reachable.
- If disk cache is corrupt or absent, the Pi enters an error state (no content to display)
  and continues polling. It does NOT reconstruct state from any other source.
- Cache freshness: if disk cache age > 120,000ms (`simulator/pi-appliance.js:41`), the
  Pi emits a stale manifest warning. Displaying from a stale cache is permitted.
  Displaying corrupted content is not (checksum validation required).

### Tier 4 — Ephemeral In-Memory State

**Owner:** Process-local. Destroyed on restart. Non-authoritative.

| Store | Contains | Write Owner | TTL |
|-------|----------|-------------|-----|
| Simulator fleet state | Per-screen poll status, metrics, last checksum | fake-pi.js / soak.js | Process lifetime |
| Management API state | Fleet status map | Simulator management API | Process lifetime |
| Backend rate limiter | Per-IP request counts | rateLimiter.js | 60s sliding window |
| Backend request IDs | In-flight request map | requestId.js | Request lifetime |

**Properties:**
- No durability. Loss on restart is expected and not an error.
- MUST NOT be used as the basis for any Tier 1 write.
- Tier 4 state feeds observability (metrics, logs, dashboards) but not decisions.

## 2. WRITE OWNERSHIP RULES

These rules are absolute. Violations are architectural defects.

| Rule | Statement |
|------|-----------|
| WO1 | Only the backend writes to Tier 1. No Pi, simulator, or external tool writes directly to PostgreSQL. |
| WO2 | Only the backend writes to Tier 2. Cache is a backend implementation detail. |
| WO3 | Only a Pi writes to its own Tier 3 cache. The backend does not push content to Pi storage. |
| WO4 | Tier 4 state is write-local to the process that owns it. No cross-process Tier 4 writes. |
| WO5 | Schema migrations are write operations to Tier 1 structure. They are executed only by the migration runner (docker-compose.dev-sim.yml `migrate` service) or explicit operator action per OPERATIONS.md. |
| WO6 | The backup/restore process (`scripts/backup.sh`, `scripts/restore.sh`) is the only authorised path for bulk Tier 1 state transfer. It is an operator-initiated action only. |

## 3. READ AUTHORITY RULES

| Component | May Read | MUST NOT Read |
|-----------|----------|---------------|
| Backend manifest route | Tier 1 (fallback), Tier 2 (primary) | Tier 3, Tier 4 |
| Pi appliance / fake-pi | Tier 2 via HTTP (manifest endpoint) | Tier 1 directly (no DB access) |
| Pi appliance offline mode | Tier 3 (local cache) | Tier 1, Tier 2 |
| Simulator fleet state | Tier 4 (its own memory) | Tier 1, Tier 2, Tier 3 |
| ops-check.py | Tier 2 via `/health/ready` + manifest endpoint | Tier 1 directly |
| Chaos tests | Tier 4 (metrics), Tier 2 (health probes) | Tier 1 directly |

## 4. CACHE AUTHORITY RULES

### 4.1 Tier 2 Cache Validity

A manifest_cache entry is valid if ALL of the following are true:
- `computed_at` is within the last 5 seconds (backend in-memory TTL)
- Checksum in cache matches the checksum that would result from a fresh Tier 1 computation
- The entry was written by the backend's own manifest computation pipeline

A cache entry is INVALID and MUST be recomputed if:
- `valid_until` has passed
- Any content or schedule referenced in the manifest has been modified since `computed_at`
- The DB returned a different checksum on the most recent computation

### 4.2 Tier 3 Cache Validity

A Pi's local cache entry is valid for display purposes if:
- The file is readable and not corrupt
- The checksum stored in the cache matches the checksum of the file contents

A Pi's local cache is STALE (may still display but must warn) if:
- The cache age > 120,000ms (`STALE_THRESHOLD` in pi-appliance.js)

A Pi MUST NOT display cached content if:
- Checksum validation fails (content integrity compromised)

## 5. RECONCILIATION RULES

Reconciliation is required when a Pi's displayed state diverges from Tier 1 state.

| Scenario | Detection | Resolution |
|----------|-----------|------------|
| Pi displays stale checksum after backend update | Poll succeeds but checksum differs from Tier 2 | Pi applies new manifest on next successful poll |
| Pi offline → displays Tier 3 → backend changes content | Pi comes online, polls, gets new checksum | Pi applies new manifest immediately |
| Tier 2 cache evicted → cold compute needed | Backend computes from Tier 1 | Next poll gets freshly computed manifest |
| Tier 1 content deleted while Pi offline | Pi holds Tier 3 reference to deleted content | Pi gets fallback manifest on next poll (backend returns empty schedule) |

**Reconciliation is pull-based, not push-based.** The backend does not notify Pis of
state changes. Pis discover changes on their next poll cycle (up to 15,000ms delay).
This is by design and is not a defect.

Maximum reconciliation lag: 15,000ms (one poll cycle) under normal conditions.
Maximum reconciliation lag under network outage: unbounded (screens serve Tier 3).
After network restoration: ≤ 15,000ms (one poll cycle per SYSTEM_CONTRACTS §4.1).

## 6. SPLIT-BRAIN RECOVERY

Split-brain occurs when multiple screens disagree on the current manifest checksum
in a way that cannot be explained by propagation lag.

**Split-brain definition (from SYSTEM_CONTRACTS §2 F1):**
> Screen count mismatch between backend and active polling screens
> (desync_count > 0 per `coherence.max_desync_count = 0` in thresholds.json)

### 6.1 Expected vs. Pathological Desync

| Situation | Desync Type | Duration | Handling |
|-----------|-------------|----------|---------|
| Content change propagating across fleet | Expected | ≤ 15,000ms | Normal operation |
| Network partition: subset offline | Expected | Duration of partition | Tier 3 fallback |
| OTA mid-rollout: mixed versions | Expected | Duration of ring rollout | Ring state machine handles |
| Backend serving incorrect manifest | Pathological | Persistent | Class F F1 — immediate investigation |
| Tier 2 cache serving wrong checksum | Pathological | Until cache invalidated | Class F F1 — `make clear-caches` |
| Version regression in manifest | Pathological | Persistent | Class F F2 — deploy blocker DB-5 |

### 6.2 Recovery Procedure

When desync_count > 0 persists for > 45,000ms (`coherence.max_desync_duration_ms`):

1. Run `make db-cache` — inspect manifest_cache checksums for anomalies
2. Run `make clear-caches` — force cold recompute from Tier 1
3. If desync persists after cache clear: Tier 1 state is corrupt or inconsistent
4. Run `make health-full` — confirm `/health/ready` is PASS
5. If Tier 1 is corrupt: halt deployment, invoke `scripts/restore.sh` from last known-good backup

## 7. DISPOSABLE STATE

State that can be deleted without data loss or service impact. Safe to destroy at any time.

| State | Location | Safe to Destroy? | Recovery |
|-------|----------|-----------------|---------|
| manifest_cache table contents | PostgreSQL `manifest_cache` | Yes | Next poll cold-computes from Tier 1 |
| Tier 4 in-memory fleet state | Simulator process memory | Yes | Process restart reinitialises |
| reports/*.json | Local filesystem | Yes | Re-run tests |
| soak-reports/*.json | Local filesystem | Yes | Re-run soak |
| Docker log buffers | Docker daemon | Yes (after `docker logs` flush) | New events only |

**NOT disposable (Tier 1):**
- `content` table rows
- `schedules` table rows
- `screens` table rows
- `venues` table rows
- `playlists` table rows
- `backups/*.sql.gz` — these ARE the recovery path; do not delete

## 8. CONFLICT RESOLUTION HIERARCHY

When two sources disagree on the value of the same piece of state:

```
1. PostgreSQL Tier 1 (written by backend)    — WINS over everything
2. manifest_cache Tier 2 (derived from Tier 1) — WINS over Tier 3 and Tier 4
3. Pi disk cache Tier 3                        — WINS only when Tier 1/2 unreachable
4. In-memory / derived state Tier 4           — NEVER wins; observation only
```

This hierarchy is consistent with Truth Hierarchy §5 of SYSTEM_CONTRACTS for the
data plane. Note: the Truth Hierarchy in §5 applies to test evidence authority.
This document's hierarchy applies to live state authority.

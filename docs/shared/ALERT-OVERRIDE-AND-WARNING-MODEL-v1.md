# ALERT-OVERRIDE-AND-WARNING-MODEL-v1

**Document class:** Implementation-grade operational information model
**Authority:** Constitutional for Override records; operational for Alert and Warning records
**Audience:** Backend API teams implementing signal delivery, override management, and PRE integration; frontend teams implementing notification surfaces and override controls

---

## Signal Classification Hierarchy

Five classes of operational signal, ordered by operational urgency. Class determines delivery surface, suppression rules, dismissal authority, and audit obligations.

| Class | Name | Generated by | Suppressible | Auto-expires | Operator dismissal |
|-------|------|-------------|-------------|-------------|-------------------|
| 1 | Constitutional Alert | System | No | No | No |
| 2 | Incident Alert | System | No | No (while incident active) | No |
| 3 | State Transition Warning | System | Collapses to summary | After 60 seconds without interaction | Yes (Level 3 dismissal logged) |
| 4 | Advisory | System | No | No | Yes (dismissal logged; resurfaces on condition worsening) |
| 5 | Informational Notice | System | N/A | N/A | No (audit trail only; no operator-visible delivery) |

### Class 1 — Constitutional Alert

Generated when the platform enters or exits a constitutional state change of highest severity. Cannot be suppressed by any operator action. Cannot be dismissed. Requires explicit operator acknowledgement.

**Triggering conditions:**
- Platform enters EMERGENCY_FREEZE
- Platform enters PRE_DISABLED
- S1 incident declared (implicit or explicit)
- COMMANDER_LAPSED alert fires after 15-minute threshold

**Delivery:** System Status Bar (highest zone). Persists until underlying condition resolves.

### Class 2 — Incident Alert

Generated on incident state machine transitions at S1–S3 severity. Cannot be dismissed while the incident remains active.

**Triggering conditions:**
- S1–S3 incident declared
- S1–S3 incident transitions to COMMANDER_LAPSED
- S1–S3 incident transitions to CONTAINED
- S1–S3 incident resolved

**Delivery:** System Status Bar plus incident panel. Auto-resolves when incident resolves.

### Class 3 — State Transition Warning

Generated on venue player state changes. Auto-expires after 60 seconds without operator interaction. Multiple Class 3 signals within a 60-second window on the same scope collapse to a fleet summary.

**Triggering conditions:**
- Venue machine state: LIVE → OFFLINE
- Venue machine state: LIVE → INCIDENT
- Venue machine state: LIVE → DEGRADED
- Venue machine state: SYNCING → OFFLINE (during sync)

**Collapse rule:** If N > 1 Class 3 signals arrive within 60 seconds on the same fleet scope, they are collapsed to a single fleet summary signal with `collapsed_count: N`. Individual signals are preserved in the audit trail but only the summary is delivered to the operator surface.

### Class 4 — Advisory

System-generated observations of approaching risk thresholds. Delivered to attention zone only (not System Status Bar). Operator may dismiss. Dismissal is audited. If the condition worsens after dismissal, the advisory resurfaces.

**Triggering conditions:** See Warning Types in the Warning Model section.

### Class 5 — Informational Notice

Audit trail only. Never surfaced to operators as a UI element. Accessible via audit trail query interface. No acknowledgement required. No delivery tracking.

## Override Model

An override is a PRE-modifying instruction placed by an authorized operator. It modifies the content resolution output for a defined scope for the duration of its active state.

### Authority by Level

| Level | PRE hierarchy | Minimum role | Session elevation required | Approval required | Confirmation required |
|-------|--------------|-------------|--------------------------|-------------------|-----------------------|
| L1 | Operational override | OPERATOR | No | No | No |
| L2 | Scheduled | OPERATOR | No | No | No |
| L3 | Campaign | OPERATOR | No | Yes (non-ADMIN) | No |
| L4 | Sponsorship | OPERATOR | Yes | Yes | No |
| L5 | Structural | ADMIN | No | Yes | No |
| L6 | Emergency | OPERATOR (elevated) or ADMIN | Yes (OPERATOR path) | No | String literal: `"EMERGENCY"` |

**Session elevation:** An elevated session is a time-bounded authentication upgrade, not a role change. Elevation expires. L4 placements require an active elevated session at the time of API call; an expired elevation is a rejected placement, not a queued one.

**Approval queue:** L3–L5 approvals are routed to any online ADMIN. If no ADMIN is online, the override remains in PENDING_APPROVAL until approval is received or the placement is withdrawn. L3 non-ADMIN placements may also be approved by ADMIN. L6 bypasses the approval queue entirely — it is immediate.

### Override Object Schema

#### Required Attributes

| Field | Type | Mutability | Constraints |
|-------|------|-----------|-------------|
| `override_id` | string | IMMUTABLE | Globally unique |
| `placed_at` | governed_timestamp | IMMUTABLE | Governed clock only |
| `placed_by` | operator_id | IMMUTABLE | |
| `level` | integer 1–6 | IMMUTABLE | PRE hierarchy level |
| `scope_id` | venue_id or fleet_id | IMMUTABLE | |
| `content_ref` | content_id or playlist_id | MUTABLE (ADMIN) | The content instruction |
| `governed_timestamp` | governed_timestamp | IMMUTABLE | Clock-governed placement time used in PRE resolution determinism |

#### Optional Attributes

| Field | Type | Notes |
|-------|------|-------|
| `expires_at` | governed_timestamp | Required for L5. Advisory-only for L6. Optional for L1–L4, but absence generates WARNING: NO_EXPIRY_OVERRIDE. |
| `expiry_warning_at` | governed_timestamp | When to generate an approaching-expiry Class 4 advisory. |
| `reason` | string | Operator-provided context. No format enforcement. |
| `approval_id` | approval_id | Reference to the approval record. Set by system on approval. |
| `parent_override_id` | override_id | When placed in direct response to another override. |
| `incident_id` | incident_id | Automatically set when an active incident exists on the scope at placement time. Not operator-settable. |

### Override Lifecycle

#### State Definitions

| State | PRE effect | Entry condition | Exit conditions |
|-------|-----------|----------------|-----------------|
| PENDING_APPROVAL | None | L3–L4 non-ADMIN placement | ACTIVE (on approval), WITHDRAWN (placement withdrawn before approval) |
| ACTIVE | Full | Placement (L1–L2, L5–L6) or approval (L3–L4) | EXPIRED, REMOVED, SUPERSEDED |
| EXPIRED | None | `expires_at` reached (system event) | Terminal |
| REMOVED | None | Authorized operator removes | Terminal |
| SUPERSEDED | None (PRE contribution = 0) | Higher-level override on same scope becomes ACTIVE | ACTIVE again (if superseding override is removed before this one expires) |

#### SUPERSEDED State Semantics

SUPERSEDED is not terminal. A superseded override is still an active record. Its `current_state` is SUPERSEDED. Its PRE contribution is zero while the higher-level override is active. If the higher-level override is removed (or expires), the superseded override's PRE contribution resumes — provided it has not reached its own `expires_at`.

**Implementation requirement:** PRE must evaluate the full override stack on every resolution, not cache the last non-superseded result. The stack is ordered by level; ties within a level are resolved by `placed_at` (most recent wins).

#### Lifecycle Diagram

```
L3-L4 non-ADMIN:
  PENDING_APPROVAL → ACTIVE → EXPIRED (auto)
                            → REMOVED (authorized operator)
                            → SUPERSEDED → ACTIVE (if superseding override removed)
                    → WITHDRAWN (before approval)

L1-L2, L5-L6:
  ACTIVE → EXPIRED (auto, if expires_at set)
         → REMOVED (authorized operator)
         → SUPERSEDED → ACTIVE (if superseding override removed)
```

### Expiry Rules

| Level | `expires_at` required | Auto-expiry behavior | Warning if absent |
|-------|----------------------|---------------------|-------------------|
| L1 | No | Expires at `expires_at` if set | Yes — NO_EXPIRY_OVERRIDE warning |
| L2 | No | Expires at `expires_at` if set | Yes — NO_EXPIRY_OVERRIDE warning |
| L3 | No | Expires at `expires_at` if set | Yes — NO_EXPIRY_OVERRIDE warning |
| L4 | No | Expires at `expires_at` if set | Yes — NO_EXPIRY_OVERRIDE warning |
| L5 | **Yes** — platform rejects L5 without `expires_at` | Expires at `expires_at` | N/A (required) |
| L6 | No | `expires_at` treated as advisory reminder, not enforcement | No — L6 is emergency |

**What may automatically expire:** L1–L5 overrides with `expires_at` set. Auto-expiry is written as a SYSTEM event in the audit trail with `expired_by: system`.

**What may never automatically expire:** L6 overrides. The `expires_at` field on an L6 override, if present, generates a Class 4 advisory at that time but does not change override state. L6 remains ACTIVE until explicitly removed.

**Exception — investigation-linked overrides:** Any override that is the subject of an active investigation (linked via `override_id` in an open INVESTIGATION record) may still be removed, but removal generates a Class 3 warning: "This override is linked to an active investigation." The removal is not blocked. The warning is audited.

### Override Accumulation Rule

When the count of ACTIVE overrides on a single scope exceeds 3, a Class 4 Advisory is generated:

> "Override accumulation — [N] active overrides on [venue/fleet name]."

The advisory persists until the count drops below 3. It does not block further override placements. Multiple advisories for the same scope are collapsed — one advisory per scope, updated with current count.

### Removal Authority

| Level | Who may remove | Confirmation required |
|-------|---------------|----------------------|
| L1 | Placed-by operator OR ADMIN | No |
| L2 | Placed-by operator OR ADMIN | No |
| L3 | Placed-by operator OR ADMIN | No |
| L4 | Placed-by operator OR ADMIN | No |
| L5 | ADMIN only | No |
| L6 | OPERATOR+ with active elevated session, OR ADMIN | String literal: `"CONFIRM REMOVAL"` |

### Override and Incident Relationship

An override placed while an active incident exists on the scope is automatically annotated with `incident_id`. This is informational — the override is not subordinate to the incident.

When the incident is resolved:
- Overrides placed during the incident are **not automatically removed**
- Post-incident override review is an operator responsibility
- The handoff package surfaces active overrides that were placed during the incident
- The RESOLVED transition does not change any override state

This is a deliberate design: operators placed overrides as tactical responses. Automatic removal on incident resolution would remove content governance decisions without operator review.

### Audit Obligations for Overrides

| Event | Trigger | Required fields |
|-------|---------|-----------------|
| OVERRIDE_PLACED | Placement | override_id, placed_by, level, scope_id, governed_timestamp |
| OVERRIDE_APPROVAL_REQUESTED | L3–L5 approval queue entry | override_id, requested_by |
| OVERRIDE_APPROVED | Approval granted | override_id, approved_by |
| OVERRIDE_ACTIVATED | Transition to ACTIVE | override_id, activation_type (immediate or post-approval) |
| OVERRIDE_EXPIRED | Auto-expiry | override_id, expired_by: system |
| OVERRIDE_REMOVED | Explicit removal | override_id, removed_by, confirmation if L6 |
| OVERRIDE_SUPERSEDED | Supersession | override_id, superseded_by_override_id |
| OVERRIDE_RESUMED | Resumed from SUPERSEDED | override_id, resumed_because_override_id |
| OVERRIDE_WITHDRAWN | Withdrawn from PENDING_APPROVAL | override_id, withdrawn_by |

## Warning Model

Operational Warnings are system-generated observations that a threshold is being approached or crossed. They are distinct from incidents. They do not require incident declaration to resolve. They may escalate to incidents, but escalation is an operator action, not automatic (except for the single PRE_DIVERGENCE exception defined below).

### Warning Types Reference

| Type | Trigger condition | Auto-resolves | May be dismissed |
|------|-----------------|---------------|-----------------|
| OVERRIDE_ACCUMULATION | >3 active overrides on a scope | Yes — when count drops below 3 | No |
| CERTIFICATION_EXPIRY | Certification expires in <30 days | Yes — on renewal or expiry | Yes (logged) |
| CORPUS_SYNC_OVERDUE | Corpus not synced within configured window | Yes — on successful sync | Yes (logged) |
| AUTONOMY_THRESHOLD | Venue offline, <24h autonomy remaining (after >1h offline) | Yes — on reconnect or corpus sync | No |
| NO_EXPIRY_OVERRIDE | L1–L4 override placed without `expires_at` | When override gains `expires_at` or is removed | No |
| ANNOTATION_ABSENCE | No operator annotation on active incident in >2 hours | Yes — on annotation written | Yes (logged) |
| COMMAND_DURATION | Same operator held incident command >12 hours | Yes — on command transfer | Yes (logged) |
| PRE_DIVERGENCE | PRE replay produces different output than original resolution | When ADMIN classifies and closes | No |

### Warning Lifecycle

```
ACTIVE → RESOLVED (condition cleared — system event)
ACTIVE → ESCALATED (operator escalates to incident, OR automatic escalation for PRE_DIVERGENCE)
ACTIVE → ACKNOWLEDGED_AND_DISMISSED (operator explicit dismissal)
```

Warnings in ESCALATED or ACKNOWLEDGED_AND_DISMISSED states are terminal.

**Warnings that may never be auto-resolved:**

| Warning type | Resolution requirement |
|-------------|----------------------|
| NO_EXPIRY_OVERRIDE | Operator adds `expires_at` to the override, or removes the override |
| PRE_DIVERGENCE | ADMIN must classify the divergence (EXPECTED, UNEXPECTED, ACKNOWLEDGED_RISK) and explicitly close the warning |

### Warning Dismissal Rules

| Warning type | May be dismissed | Condition resurfaces |
|-------------|-----------------|---------------------|
| OVERRIDE_ACCUMULATION | No | — |
| CERTIFICATION_EXPIRY | Yes | If certification expires without renewal (escalates to Class 3) |
| CORPUS_SYNC_OVERDUE | Yes | If sync continues to be overdue (interval doubles on dismissal) |
| AUTONOMY_THRESHOLD | No | — |
| NO_EXPIRY_OVERRIDE | No | — |
| ANNOTATION_ABSENCE | Yes | If absence continues another 2 hours |
| COMMAND_DURATION | Yes | If command continues another 6 hours |
| PRE_DIVERGENCE | No | — |

Dismissal is always audited with the dismissing operator's identity and governed_timestamp.

## Alert, Override, and Warning — Relationship Model

The relationships between these three signal types are explicit. Ambiguity here produces incorrect operator mental models.

### Rule 1: Alerts are generated by system events, not by overrides or warnings

An alert is triggered by a system event — a constitutional state change, an incident state machine transition, or a circuit breaker event. An override being placed is not a system event that generates an alert. A warning being generated is not a system event that generates an alert. These are distinct signal pathways.

```
System event ──► Alert (Class 1 or 2)
Override placement ──► (no alert) ──► Warning (if NO_EXPIRY_OVERRIDE or ACCUMULATION)
Warning ──► (no alert, unless PRE_DIVERGENCE auto-escalation rule applies)
```

### Rule 2: Overrides do not generate incidents

An override placement, even L6, does not declare an incident. An override may generate a warning. A warning may be escalated to an incident by an operator. The causal chain is: Override → Warning (possibly) → Operator action → Incident (possibly).

### Rule 3: Incidents generate alerts, not overrides

An S1–S3 incident declaration generates a Class 2 alert. The incident does not create overrides. Operators may place overrides in response to an incident, but this is an operator action, not an automatic system behavior.

```
Incident (S1-S3) ──► Class 2 Alert (system)
Incident ──► (no overrides created)
Operator response to incident ──► Override (operator action)
```

### Rule 4: Resolution is independent

Resolving an incident does not resolve associated overrides.
Removing overrides does not resolve an incident.
These are independent state machines with optional informational links.

### Rule 5: References are annotations, not ownership

An alert may reference an `incident_id`. An override may reference an `incident_id`. A warning may reference an `incident_id`. The incident does not own these references. They are annotations written on the alert/override/warning record. The incident record does not contain alert_ids, override_ids, or warning_ids. These are derived by querying the alert/override/warning records.

## Escalation Behavior

### Class 3 → Class 2 (Warning to Incident Alert)

A warning is escalated to an incident when an OPERATOR+ declares an incident referencing the warning as origin. The warning transitions to ESCALATED. The resulting incident record contains the warning_id in its `origin_warning_id` field (optional).

The escalation creates the incident — it does not change the alert class directly. The incident declaration triggers a Class 2 alert as a result of the incident existing.

### Class 4 → Class 3 (Advisory to State Transition Warning)

An ADMIN may upgrade an advisory to a Class 3 state transition warning. Non-ADMIN operators cannot promote advisory signals. This is used when a threshold observation warrants more prominent delivery before an operator has escalated to a full incident declaration.

### Automatic Escalation (PRE_DIVERGENCE only)

The only permitted automatic alert escalation without operator action:

When a PRE_DIVERGENCE warning is active and the `parity_ratio` (the proportion of replay executions producing matching output) falls below the constitutional threshold (defined in `thresholds.json`), the system may automatically generate a Class 2 alert. This alert does not create an incident — it escalates the signal class only.

The audit event for this automatic escalation must include: `escalation_type: AUTOMATIC`, `trigger: parity_ratio_threshold`, `observed_ratio`, `threshold`.

All other escalations require an OPERATOR+ action.

## Advisory and Informational Notice Detail

### Class 4 Advisory Delivery Rules

- Delivered to attention zone only (Zone C per attention governance specification)
- Does not appear in System Status Bar
- Does not interrupt primary workflow surfaces
- Operator may dismiss; dismissal is logged
- Dismissed advisories resurface if the underlying condition worsens — specifically:
  - CERTIFICATION_EXPIRY dismissed at 30 days resurfaces at 7 days and at 1 day
  - CORPUS_SYNC_OVERDUE dismissed resurfacess after double the original overdue interval
  - COMMAND_DURATION dismissed at 12 hours resurfaces at 18 hours
  - ANNOTATION_ABSENCE dismissed at 2 hours resurfaces at 4 hours

### Class 5 Informational Notice Rules

Class 5 signals are never delivered to any operator-visible surface. They are:
- Written to the audit trail with full provenance
- Queryable via the audit trail interface
- Usable for debugging and forensic replay
- Not counted in any operator-facing notification count

## Audit Obligations — All Signal Classes

Every signal, regardless of class, must be written with the following fields. A signal without a traceable `source_event_id` is a data integrity violation. The API must reject or flag signal writes without this field.

### Mandatory fields on every signal record

| Field | Type | Notes |
|-------|------|-------|
| `signal_id` | string | Globally unique |
| `generated_at` | governed_timestamp | |
| `signal_class` | integer 1–5 | |
| `source_event_id` | event_id | The specific system event that triggered generation. Non-null. Non-empty. |
| `scope_id` | venue_id or fleet_id | |

### Per-operator delivery tracking

Stored separately from the signal record. Each operator has independent delivery state.

| Field | States |
|-------|--------|
| `delivery_state` | DELIVERED, ACKNOWLEDGED, DISMISSED, EXPIRED |
| `delivered_at` | governed_timestamp |
| `acknowledged_at` | governed_timestamp or null |
| `dismissed_at` | governed_timestamp or null |

### Audit events per signal class

| Class | Events written |
|-------|---------------|
| 1 | SIGNAL_GENERATED, SIGNAL_DELIVERED, SIGNAL_ACKNOWLEDGED (per operator), SIGNAL_RESOLVED |
| 2 | SIGNAL_GENERATED, SIGNAL_DELIVERED, SIGNAL_ACKNOWLEDGED (per operator), SIGNAL_RESOLVED |
| 3 | SIGNAL_GENERATED, SIGNAL_DELIVERED, SIGNAL_ACKNOWLEDGED, SIGNAL_EXPIRED or SIGNAL_COLLAPSED |
| 4 | SIGNAL_GENERATED, SIGNAL_DELIVERED, SIGNAL_ACKNOWLEDGED, SIGNAL_DISMISSED (if dismissed), SIGNAL_RESURFACED (if condition worsens) |
| 5 | SIGNAL_GENERATED only (no delivery events) |

## Override Stack — PRE Integration Contract

PRE evaluates the override stack on every content resolution. The override stack is the ordered set of all ACTIVE or SUPERSEDED overrides on a given scope, sorted by level (highest first) and then by `placed_at` (most recent first within a level).

**PRE stack evaluation rules:**

1. Only ACTIVE overrides contribute to PRE output. SUPERSEDED overrides have zero contribution.
2. The highest-level ACTIVE override on the scope wins for content selection.
3. Ties (two ACTIVE overrides at the same level on the same scope) are resolved by `placed_at` — most recent wins.
4. If no override is ACTIVE on the scope, PRE resolves via its standard level hierarchy.

**Determinism requirement:** Given the same set of override records (same IDs, states, timestamps) and the same `governed_timestamp` for the resolution query, PRE must produce the same output. Override stack state at any `governed_timestamp` must be reconstructible from the audit trail.

**PRE must not cache the override stack between resolution calls.** The stack is computed fresh at each resolution. Caching is a determinism risk.

## API Contract Summary

### Override Endpoints

| Endpoint | Method | Authority | Notes |
|----------|--------|-----------|-------|
| `/overrides` | POST | OPERATOR+ (level-dependent) | Body includes level, scope_id, content_ref, optional expires_at, reason |
| `/overrides/{id}` | GET | OPERATOR+ (scope access) | Returns full override record with current state |
| `/overrides/{id}/approve` | POST | ADMIN | L3–L5 approval |
| `/overrides/{id}` | DELETE | Role-dependent (see removal authority table) | Confirmation body required for L6 |
| `/overrides/stack/{scope_id}` | GET | OPERATOR+ | Returns ordered override stack for PRE diagnostic purposes |

### Alert Endpoints

| Endpoint | Method | Authority | Notes |
|----------|--------|-----------|-------|
| `/alerts` | GET | OPERATOR+ (scoped) | Returns alerts for operator's scopes |
| `/alerts/{id}/acknowledge` | POST | OPERATOR+ | Per-operator acknowledgement |
| `/alerts/{id}/dismiss` | POST | OPERATOR+ | Level 3–4 only; 1–2 rejected |

### Warning Endpoints

| Endpoint | Method | Authority | Notes |
|----------|--------|-----------|-------|
| `/warnings` | GET | OPERATOR+ (scoped) | Returns active warnings for operator's scopes |
| `/warnings/{id}/dismiss` | POST | OPERATOR+ | Dismissible types only; non-dismissible rejected |
| `/warnings/{id}/escalate` | POST | OPERATOR+ | Creates incident referencing this warning |
| `/warnings/{id}/classify` | POST | ADMIN only | PRE_DIVERGENCE classification and closure |

**No POST `/alerts` endpoint.** Alerts are system-generated. Client-originated alert creation is rejected with 405.
**No POST `/warnings` endpoint.** Warnings are system-generated. Client-originated warning creation is rejected with 405.
